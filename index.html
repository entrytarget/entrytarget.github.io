Entry Target

A stateless financial ledger API. High performance. Uncompromising consistency.

Financial infrastructure should not be a source of uncertainty. Entry Target is a double-entry ledger delivered as an API — purpose-built for teams that require correctness, auditability, and throughput. The architecture is minimal by design. A proven accounting model, executed with precision.

What is it

Entry Target is a financial ledger API. You create accounts, register financial transfers between them, and retrieve balances. Credits equal debits. The books close on time.

The architecture is conservative by design — a Rust monolith communicating directly with the database. It runs on your AWS account. One container on Fargate, one Postgres instance on RDS. Your data never leaves your infrastructure.

The result is an API that handles millions of transactions with full consistency: ACID-compliant, double-entry, immutable audit trail.

Features

Double-entry bookkeeping

Every transaction produces a debit and a credit — or a batch of them. Always balanced. Always consistent. The double-entry method is the established standard for accounting, and this system implements it without deviation.

Immutable audit trail

Once a transaction is recorded, it remains recorded. No silent modifications. No unaccounted deletions. Corrections are made through compensating entries, following standard accounting practice.

Integrity verification

Call GET /integrity to verify the entire database. Every row is validated. If any record was altered directly in Postgres — whether by an administrative error, an unauthorized script, or a misconfigured process — the system detects it. Accounts with integrity violations are immediately blocked from further transactions. Immutability is enforced at the application level.

Multi-asset support

The system tracks assets, not currencies specifically. Whether the unit of account is dollars, euros, or inventory items, the ledger handles it with arbitrary precision. Each journal entry declares its asset, and every account involved must match. Multiple assets coexist within the same ledger without constraint.

No silent deduplication

Duplicate requests are not silently absorbed. If the same transaction is submitted twice, the system rejects it explicitly. Deduplication is a responsibility of the calling system. The ledger treats every request as intentional, ensuring that integration errors are detected at the point of origin.

High performance

Thousands of transactions per second on modest infrastructure. The system achieves this through batched writes, minimal abstraction layers, and direct database communication.

Journal archiving

The journal grows by design. As volume increases, query performance over older records will degrade. The archive endpoint allows you to select a cutoff date and compact older entries into dedicated storage. Recent entries remain in the active journal. You control the definition of "recent," the timing, and the frequency.

Metrics endpoint

A GET /metrics endpoint is available for Prometheus to scrape. Connect Grafana, configure dashboards, and establish alerts. Observability is included as a standard capability of the system.

Catastrophic recovery

If a batch was committed and the process terminated before delivering the response, the outcome is ambiguous. GET /recovery returns every transaction from the last committed batch after a restart. This eliminates orphaned transactions and the need for manual log analysis.

Balance by category

Query all account balances for a given asset, grouped by account category, in a single call. This provides the data necessary to construct a real-time balance sheet or any other consolidated view.

Balance policies

When posting a transaction, you specify — for each account involved — whether its balance must remain positive, must remain negative, or has no restriction. Business logic remains in the calling system. The ledger enforces the arithmetic constraints declared in each request.

Simple API

REST. JSON. Four primary endpoints. The API surface is minimal, and the documentation reflects that.

Creating a transaction:

POST /v1/transactions
{
  "entries": [
    {
      "sequence": 1,
      "type": "SOME_PAYMENT_METHOD",
      "asset": "USD",
      "amount": 500,
      "debit": {
        "account_id": "debit-account-uuid",
        "balance_policy": "ALWAYS_POSITIVE"
      },
      "credit": {
        "account_id": "credit-account-uuid",
        "balance_policy": "NONE"
      }
    }
  ]
}

A complete request requires only the fields shown above.

FAQ

What technology is used?

The system is written in Rust, a language designed for correctness, memory safety, and performance. The database is PostgreSQL — a mature, battle-tested relational database. Communication is over standard HTTP.

For a system where accounting precision is non-negotiable, a language that enforces correctness at the compiler level is a natural choice.

Is AWS required?

At present, yes. The system runs on AWS Fargate and RDS Postgres. Support for additional cloud providers is on the roadmap.

Can this be used for digital assets or alternative currencies?

The system tracks any financial transaction regardless of the asset type. It ensures the numbers balance correctly. The classification of what is being accounted for is determined by the calling system.

What throughput can be expected?

With all entries operating on the same balance: 21,000 transactions per second on a local machine. 3,000 TPS on the smallest available Fargate and RDS instances. Higher throughput is achievable with larger infrastructure configurations. For workloads exceeding 3,000 TPS, we recommend contacting our team to discuss the appropriate setup.

architecture

A deployment model built for operational clarity.

EntryTarget keeps the operating model straightforward: the runtime and deployment workflow are provided by the platform, while the ledger data and infrastructure boundary remain in the customer's AWS account.

what lives in your account

Operational control stays with the team that owns the risk.

The deployed footprint is explicit and reviewable before it runs.

  • Fargate service running the customer-dedicated ledger runtime.
  • RDS Postgres selected by your team, sized on your terms.
  • CloudWatch log groups and runtime heartbeat signals.
  • Secrets Manager entries for runtime secrets and credentials.
  • Customer region, VPC constraints and account-level ownership.
what stays with entrytarget

The platform handles workflow, releases and operator tooling.

The model is intentionally straightforward so finance, platform and security teams can reason about the boundary without guesswork.

  • Environment creation, billing state and console workflow.
  • IAM template generation and region-aware infrastructure scaffolding.
  • Release packaging, deployment jobs and upgrade guidance.
  • Support for the runtime, not ownership of transactional records.

product surface

A focused runtime built around the surfaces teams actually operate.

The platform is intentionally compact. It concentrates on the parts that matter most in production: journal correctness, operational visibility, integrity and recovery.

journal

Double-entry recording without improvisation.

Record balanced transactions with explicit debit and credit entries. The model stays legible to finance teams instead of disappearing into application-side bookkeeping logic.

POST /v1/transactions

{
  "entries": [
    { "account": "cash", "type": "debit" },
    { "account": "revenue", "type": "credit" }
  ]
}

-> balanced before commit
balances

Real-time balances without the reporting detour.

Account state is queryable directly instead of reconstructed from export pipelines. That makes operational checks faster and fewer things have to agree by accident.

environment

Account and region are first-class choices.

The operator flow starts by choosing a friendly environment name, a 12-digit AWS account and the target region. That keeps deployment ownership explicit from the first screen.

billing

Trial is generous, paid plan is plain.

One lifetime trial environment gets full access for 30 days. After that, the paid path is a flat $250 per month plus AWS usage, not a transaction-indexed tax.

access

IAM access is generated, scoped and reviewable.

The deployment workflow can provide a region-aware CloudFormation template so teams can review the permission boundary instead of handing over broad AWS access by email or guesswork.

runtime tuning

Infrastructure profile is part of the contract.

Fargate CPU and memory, RDS class, read replicas, batch size, timeout and pool size are operator decisions surfaced in the product instead of hidden behind support tickets.

risk and security

Security model grounded in explicit boundaries.

The strength of the model is that the deployment boundary, secret handling and recovery path are concrete enough for engineering and security teams to review directly.

boundary

Customer-isolated runtime

Every environment maps to a dedicated runtime in the customer's AWS account. That removes the shared ledger plane and most of the custody ambiguity that comes with it.

secrets

Credential handling that stays legible

AWS credentials are validated and encrypted at rest. API keys are not stored in plain text. The system design is optimized for operator clarity, not just feature checkboxes.

signals

Recovery and heartbeat are visible states

Heartbeat coverage, deployment logs and recovery signals are surfaced so operators can see when the runtime is healthy, degraded or needs action.
